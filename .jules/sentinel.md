## 2025-02-27 - [Authorization Bypass and Silent Failure in User Deletion]
**Vulnerability:** The `DeleteUser` function in `controller/user.go` had two critical issues. First, the authorization check `if myRole <= originUser.Role` locked out the Root user from deleting other Root users or accounts with equal/higher role values. Second, if `model.HardDeleteUserById(id)` failed, the endpoint responded with a HTTP 200 OK and `{"success": true}`, masking the failure.
**Learning:** Hardcoded privilege checks using simple comparison operators (`<=`) can inadvertently lock out the highest-level administrators if exceptions are not made. Additionally, error handling in Gin controllers must always translate internal errors to proper API errors rather than defaulting to hardcoded success payloads, which creates a silent failure anti-pattern and desynchronizes client state.
**Prevention:** Always append `&& myRole != common.RoleRootUser` when restricting access based on role comparisons to ensure supreme administrative control is maintained. Ensure all database operations that can fail use `common.ApiError(c, err)` instead of returning hardcoded success payloads.
